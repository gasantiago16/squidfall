1 · What Squidfall is

A containerized, agentic AI chat application. The shipped example is a weather assistant: a user chats in a web UI, an LLM agent decides when it needs real-world data, it calls tools over MCP to geocode a place and fetch a forecast from api.weather.gov, and chat history can persist to PostgreSQL. Everything runs as 5 Docker containers orchestrated by a Makefile + compose.yml using Compose profiles.

3 · The 5 services

1 · database

alpine 3.23 + Postgres:5432

First-boot initdb, listens on *, MD5 auth, named volume squidfall_database. Profiles: all, backend, database.

⚠ host all all 0.0.0.0/0 md5 is wide-open — local only; tighten before deploy.

2 · backend

Django + ninja:8000

chats app: Chat(session_id, message); API GET /{id}, GET /, POST /. DB_ENGINE=postgres → Postgres, else SQLite (so CI needs no DB). Profiles: all, backend.

3 · tools

FastMCP:8002 /mcp

MCP server over streamable-HTTP. Tools: get_coordinates (geocode.maps.co), get_forecast (api.weather.gov). Needs GEOCODING_API_KEY. Profiles: all, inference, tools.

4 · inference

LangGraph ReAct + AG-UI:8001

ReAct agent served over AG-UI (ag-ui-langgraph); local Ollama qwen2.5 via ChatOllama on the native API (not /v1); MCP tools from TOOLS_ENDPOINT. Profiles: all, inference.

Compiled with an InMemorySaver checkpointer (AG-UI calls aget_state() each run). Needs langchain ≥ 1.2.

5 · frontend

Next.js 16 + CopilotKit:80 → :3000

App Router/TS/Tailwind with a liquid-glass UI (aurora + frosted glass, from the Walking Trader terminal). /api/copilotkit → LangGraphHttpAgent → squidfall-inference:8001/api/v1. Standalone output. Profiles: all, frontend.

4 · Orchestration

Makefile / sf.ps1 (the PowerShell wrapper, since make isn't installed here) are the control surface — build / start / stop / status with DOCKER_COMPOSE_PROFILE selecting scope and COMPOSE_BAKE=true for parallel builds. compose.yml defines the 5 services + volume; profiles: gate which come up; database has a healthcheck and backend waits on it.

./sf.ps1 build database     # build just the db image
./sf.ps1 start database     # bring it up
./sf.ps1 start all          # bring up the whole stack

5 · CI/CD we are adding (the blank doc pages)

The published CI, CD, Platform Resources, Deployment, Documentation pages are empty stubs (Step 1. Text goes here.). We design and build them locally.

CI — ✅ live (ci.yml)

Build 5 images → Django check+tests on SQLite → tools/inference import checks → ruff (critical) → Trivy (report-only). Ephemeral docker run (no compose up) to avoid collisions.

CD — merge / tag

Tag images by SHA → push to local registry → deploy via make ... start → post-deploy health gate → rollback on failure.

Local git runner — live

GitHub repo + self-hosted Actions runner. Repo gasantiago16/squidfall; runner squidfall-win (runs-on: [self-hosted, windows, x64]) on our hardware. GitHub triggers; build/scan/deploy run locally to limit exposure.
